# 工厂模式

工厂模式，细分为三种：简单工厂、工厂方法、抽象工厂。 它属于创建型的模式，Java 最常用的设计模式之一，帮助我们创建对象，而忽略其中具体创建逻辑。

# 简单工厂 (Simple Factory)

  • 根据 type 来创建不同的对象，常和策略模式搭配使用。

# 工厂方法模式

定义一个创建对象的接口，让子类决定具体实例化哪一个类，工厂方法使一个类的实例化延迟到子类。

  • 从例子里上看，对象的创建全是由工厂去处理。而不需要知道对象创建的具体流程，它将对象的使用和实现中进行了解藕。

  • 它可以由此创建一个复杂对象。例如在手机里面，有各式各样的零件 (对象的属性)，这些当然不能由消费者自己去组装。而工厂就可以将这些零件组装成一个完整的手机 (对象)，我们只需要告诉它，想要的是哪种即可。

优点：

  1. 屏蔽具体的实现细节，使对象的创建和使用进行了解藕
  2. 扩展性高，相较于简单工厂，他产品的增 / 删，并不需要修改原有的工厂代码，满足开闭原则。

缺点： 每新增一个产品，都需要增加两个相应的子类，即产品的实现、产品工厂的实现，类成倍的增加，会相应的提供系统的复杂度。

和简单工厂的对比：如果一个产品数量固定较少且构建简单时，可以使用简单工厂。而当这个对象复杂，且数量不固定时，可以使用工厂方法，它可以将具体的实现，交给子类。而非像简单工厂一样，只能在一个类中处理，这样会导致工厂类也会变得十分复杂。

# 抽象工厂

抽象工厂为产品家族提供接口。通过抽象工厂所提供的接口，可以创建产品的家族，复用这个接口书写代码，我们的代码将从实际工厂解耦，以便在不同上下文中实现各式各样的工厂，制造出各种不同的产品。 抽象工厂模式提供一个接口，用于创建相关或依赖对象的家族，而不需要明确指定具体类。 抽象工厂允许客户使用抽象的接口来创建一组相关的产品，而不需要知道 (或关心) 实际产出的具体产品是什么。这样一来，客户就从具体的产品中被解耦。 抽象工厂的方法经常以工厂方法的方式实现。 (引用 Head First 设计模式)

# 角色

AbstractFactory：抽象工厂，声明产品系列的创建方法 ConcreteFactory：具体工厂，创建对应的产品系列 AbstractProduct：抽象产品 / 接口，定义同类的产品，如 cpu ConcreteProduct：具体的产品实现 Client：客户端，获取对应的产品对象

优点：

  1. 抽象工厂模式隔离了具体类的生成，使得客户并不需要知道什么被创建。由于这种隔离，更换一个具体工厂就变得相对容易，所有的具体工厂都实现了抽象工厂中定义的那些公共接口，因此只需改变具体工厂的实例，就可以在某种程度上改变整个软件系统的行为。

  2. 当一个产品族中的多个对象被设计成一起工作时，它能够保证客户端始终只使用同一个产品族中的对象。

  3. 增加新的产品族很方便，无须修改已有系统，符合 “开闭原则”。

缺点：

增加新的产品等级结构麻烦，需要对原有系统进行较大的修改，甚至需要修改抽象层代码，这显然会带来较大的不便，违背了 “开闭原则”。 和工厂方法模式不同，抽象工厂提供的是一系列产品的创建，工厂方法提供的是单个产品的创建。
